# ⭐⭐⭐½ 别教模型搜索：可验证搜索无法蒸馏为思维链

**日期**: 2026年6月24日

---

论文 : A Verifiable Search Is Not a Learnable Chain-of-Thought链接 : https://arxiv.org/abs/2606.21884我们常以为，只要能把解题步骤写成自然语言，大模型就能通过监督微调（Supervised Fine-Tuning, SFT）学会这种“思维链”。这篇论文用一组受控实验狠狠打脸了这个直觉： 可验证的搜索过程，根本无法被蒸馏成模型的可学习推理路径。 这不仅是理论探讨，更是对当前 RAG、Agent 和微调范式的重大警示。
### 为什么“能搜”不等于“能学”？
作者构建了一个包含 9 个确定性生成任务的基准测试。这些任务由 Python 求解器驱动，且训练集与测试集来自同一分布，这意味着我们可以精准测量“程序逻辑”向“模型推理”迁移的效率。
核心发现令人震惊： 求解器的准确率并不能预测模型微调后的表现。
- 正向计算任务（如单位换算、罗马数字）：求解器准确率 100%，模型微调后也能达到 ≥0.99。这类任务有清晰的从左到右推导路径，模型学得很快。
- 回溯搜索任务（如算术密码 Cryptarithm）：求解器通过回溯搜索能达到 71% 的准确率，但无论怎么调整思维链设计、使用 RLVR 还是自训练，模型微调后的准确率始终卡在 0.01–0.07。
⚠️ 反直觉发现 ：模型并非算错了算术。在算术密码任务中，模型每一行的算术计算正确率高达 97–100%，且能将正确答案排在前 8 名内的概率达到 71%。它失败的原因不是“能力不足”，而是无法将搜索过程转化为线性推导。
### 核心机制：Verdict-as-Token（判决即令牌）
为什么模型学不会搜索？作者通过行级保真度审计揭示了致命缺陷： Verdict-as-Token 。
在监督微调中，模型学会了模仿“验证步骤”的 表面形式 ，却丢失了其背后的 逻辑依赖 。
- 现象：模型会枚举候选项，计算结果，然后无条件地输出“不匹配 -> 丢弃”。
- 数据证据：在算术密码任务中，虽然单行算术正确率接近 100%，但基于该行数字得出的“判决”（Verdict）正确率仅为 16–57%。
- 本质：这是一种典型的暴露偏差（Exposure Bias）。在教师强制训练时，模板总是出现在正确的上下文中；但在自由推理时，模型只是机械地重复这个“丢弃”令牌，完全脱离了当前的计算证据。

- 错误路径：试图让模型在推理时进行回溯搜索、维护尝试集合。这会导致无限循环或截断。
- 正确路径：预处理组合核心，将其转化为有限目录（Catalog）。例如，将算术密码的可能候选项预计算为 4,205 条的签名目录，模型只需执行“回忆 + 有界验证”。
这对工程实践的指导意义极其明确：
- 区分任务类型：对于需要回溯、状态维护的复杂搜索任务，SFT 蒸馏思维链是死路一条。
- 重构数据流：将搜索逻辑前置到预处理阶段，转化为模型可记忆的静态知识或检索结构，而非动态推理步骤。
- 警惕虚假推理：高算术正确率不代表高推理保真度。务必审计中间步骤的逻辑依赖性，防止模型陷入“判决即令牌”的幻觉。
### 局限与展望该结论在 3B 到 671B 的不同架构（Transformer, Mamba-MoE）上均得到验证，具有架构通用性。但这也意味着，只要 LLM 的核心生成机制是左到右的自回归，这种“搜索-推导”的不匹配就将长期存在。未来的方向可能不在于让模型更好地“搜索”，而在于如何更智能地将搜索空间压缩为模型可消化的“记忆块”。
