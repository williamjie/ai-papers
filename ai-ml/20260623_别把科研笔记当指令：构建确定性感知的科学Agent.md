论文 : Notes2Skills: From Lab Notebooks to Certainty-Aware Scientific Agent Skills链接 : https://arxiv.org/abs/2606.11897现在的 AI for Science 大多盯着发表的 Protocol 看，却忽略了实验室里最真实的资产——实验记录本（Lab Notebooks）。
这篇论文指出了一个被忽视的工程痛点： 实验笔记不是确定的指令，而是混合了事实、猜测和建议的“认知混合物”。
如果直接把笔记喂给 Agent，Agent 会把作者的犹豫当成命令，或者把确定的结论当成闲聊。这不仅是准确率问题，更是安全问题。
### 为什么现有的提取方法会翻车？
传统的方法假设输入文本是“规定性”的（Prescriptive）。比如菜谱说“混合”，Protocol 说“离心 10 分钟”。这些文本在写出来时，作者已经消除了不确定性。
但实验笔记不同。同一段话里可能包含：
- 事实 (FACT)：“读数在 5 分钟后急剧下降。”
- 判断 (JUDGMENT)：“我不确定第二次读取是否可靠。”
- 建议 (SUGGESTION)：“下次试试新鲜缓冲液。”
如果 Agent 无法区分这三者，就会发生两种致命错误：
- 不确定性清洗 (Uncertainty Laundering)：把作者的犹豫（“我不确定…”）编译成确定的执行动作。Agent 可能基于一个未解决的判断就丢弃了底层数据。
- 指令丢失 (Directive Loss)：把作者明确的指令（“这部分无效，截断它”）当成普通意见处理，导致 Agent 保留了本该排除的数据。
### Notes2Skills 的核心设计直觉论文提出的 Notes2Skills (N2S) 框架，核心 Insight 是： 作者的确定性（Certainty）必须作为 Agent 执行动作的安全边界。
它采用两阶段设计：
- 第一阶段：认知指令提取 (Epistemic Directive Extraction, EDE)。不只是提取“做什么”，还要联合预测每个指令的确定性标签（FACT/JUDGMENT/SUGGESTION）。
- 第二阶段：MetaSkill 编译。将提取结果编译成一种带有元数据的 Markdown 技能文档。每个指令都绑定了它的确定性标签和来源链接。
⚠️ 关键设计 ：确定性标签直接决定了 Agent 能做什么。FACT 授权强操作（如删除数据）；JUDGMENT 默认保守，仅保留供人工审查；SUGGESTION 仅为建议，无执行权限。
这种设计把“不确定性”从需要消除的噪音，变成了控制执行强度的信号。
### 实验结果：确定性是唯一的解法作者在三个真实湿实验室会话（Nanopore 数据）上进行了下游评估，对比了 7 种配置。结果非常硬核：

- 纯动作技能的陷阱：即使有执行器（Executor），如果技能只包含动作而不含确定性标签，执行器会因为缺乏授权信息而过度保守，导致在事实主导的场景下准确率从 88.9% 暴跌至 44.4%。
- N2S 的优势：它是唯一一种既避免了“不确定性清洗”，又没有丢失“明确指令”的配置。
### 给工程师的启示- 别把 LLM 的输出直接当代码跑。在科学发现等高风险场景，LLM 的“幻觉”或“过度自信”可能导致数据污染。你需要一个确定性的执行器（Executor）作为守门员。
- 结构化元数据比文本更重要。Notes2Skills 证明，保留作者的“认知状态”（Certainty）比单纯提取“动作序列”更有价值。在构建 Agent Skill 时，考虑加入置信度、来源链接等元数据字段。
- 确定性是安全边界。对于涉及不可逆操作（如删除实验数据）的 Agent，必须引入基于证据的对齐检查。只有当胶囊中的授权级别和信号证据都支持时，才允许执行强动作。
### 局限与展望目前该方法依赖于第一阶段的提取精度。在压力测试中，如果第一阶段由模型预测而非人工标注，指令检测的精确率会下降（例如 Step-drop 场景降至 47.9%），导致部分文件被过度保守地标记为审查。
但这恰恰指明了方向： 未来的科学 Agent 不应只是“执行者”，更应是“认知对齐者”。 只有理解科学家在笔记中留下的犹豫与确信，AI 才能真正成为可靠的共同研究者。
