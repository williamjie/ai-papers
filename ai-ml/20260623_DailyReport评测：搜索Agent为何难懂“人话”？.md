论文 : DailyReport: An Open-ended Benchmark for Evaluating Search Agents on Daily Search Tasks链接 : https://arxiv.org/abs/2606.12871现在的搜索 Agent 卷得很凶，但大多数评测集还在用那种“专家视角”的硬核科研题。这种题目在真实用户场景里几乎不会出现，导致我们很难判断 Agent 到底能不能解决老百姓的日常问题。DailyReport 的出现就是为了填补这个空白，它把评测焦点拉回了真实的、有时效性的日常搜索需求。
### 痛点：现有评测太“装”，不够接地气目前的搜索 Agent 评测基准（如 BrowseComp, DeepResearch Bench）主要存在两个工程落地痛点：
- 任务脱离实际：大多由领域专家构建，侧重专业领域的深度研究。这些任务静态且封闭，无法捕捉实时变化的用户兴趣，甚至存在数据污染风险。
- 评估缺乏可解释性：传统方法通常使用粗粒度的任务级评分规则（Rubrics），最后线性加权得出一个总分。这掩盖了 Agent 在具体子任务上的失败原因，也无法从用户视角量化“这个答案到底有没有用”。
### 核心设计：级联评分与用户中心聚合DailyReport 的核心 Insight 在于： 用户的满意度不是各个维度得分的简单平均，而是具有层级依赖关系的。
如果 Agent 连用户问的“哪所大学”都搞错了（指令遵循失败），那么后续关于该校优势的分析（事实性、合理性）对用户来说毫无意义。基于此，论文设计了以下关键机制：
- 任务构建：包含 150 个开放性问题，覆盖 10 个大类和 35 个细粒度类别。题目源自微博、Twitter 等平台的热门话题，确保时效性和真实性。
- 级联评分规则（Cascade Rubrics）：将任务拆解为原子子任务，沿三个正交维度评估：
指令遵循（Instruction Following）：是否准确理解并执行了约束？
- 事实性（Factuality）：内容是否准确？需外部搜索验证。
- 合理性（Rationality）：逻辑推理是否连贯？侧重主观分析部分。
- 用户偏好聚合（User-centric Aggregation）：引入“子任务重要性”权重。如果某个子任务缺失导致整体体验从“可用”跌到“不可用”，其权重极高。最终计算出一个 1-4 分的 User Preference Score，直接反映用户体验。

具体来看：
- 指令遵循普遍较高：几乎所有系统的 InstFollow 分数都在 0.9 以上，说明大模型理解显式约束的能力已经很强。
- 事实性是最大瓶颈：Factuality 维度得分最低。特别是在分析型任务中，由于证据分散、来源异构，模型更容易产生幻觉或引用错误信息。
- 工具调用策略差异：带搜索工具的 LLM 表现优于原生 DRA。原生 DRA 可能为了平衡成本和延迟，内部机制限制了检索深度，导致在复杂日常问题上不如通用强模型+直接搜索有效。
### 工程启示与局限对于正在构建或优化搜索 Agent 的工程师，这篇论文提供了明确的调优方向：
- 重视检索充分性：实验显示，搜索工具调用次数与整体性能正相关。不要为了省 Token 而过度复用上下文，确保在生成前进行足够的迭代检索。
- 强化引用对齐：虽然大多数模型能提供参考文献（Reference_Ratio 高），但引用的事实准确性（Reference_Support）依然偏低。优化 Reference-Claim 的对齐机制是提升事实性的关键。
- 区分任务类型：在结构化信息丰富的领域（如政治、法律），Agent 表现较好；而在动态变化、主观性强的领域（如体育、娱乐），表现较差。针对后者，可能需要更激进的多源交叉验证策略。
DailyReport 提供了一个更贴近真实世界的评测视角。它提醒我们，Agent 的进步不能仅看能否回答复杂的科研问题，更要看它能否在嘈杂的日常信息流中，给出准确、有用且符合用户直觉的答案。
