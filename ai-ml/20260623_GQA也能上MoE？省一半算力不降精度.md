# ⭐⭐⭐½ GQA 也能上 MoE？省一半算力不降精度

**日期**: 2026-06-23

---

论文 : Grouped Query Experts: Mixture-of-Experts on GQA Self-Attention链接 : https://arxiv.org/abs/2606.20945长上下文场景下，自注意力机制的二次方复杂度是扼杀推理速度的元凶。虽然 Grouped-Query Attention (GQA) 通过共享 Key-Value (KV) Head 缓解了显存压力，但它依然对每个 Token 计算所有 Query Head，这种“一刀切”的计算方式在长序列中显得极其浪费。
这篇论文提出了 Grouped Query Experts (GQE) ，一个大胆的工程直觉：既然 MoE（混合专家）能在 MLP 层通过稀疏激活大幅节省算力，为什么不能在注意力层的 Query 端也搞一套“按需分配”？
### 痛点：均匀的算力分配是低效的在标准 Transformer 中，无论 Token 是复杂的实体词还是简单的标点符号，模型都会调用所有的 Attention Head。
- 信息密度不均：实义词需要专门的 Head 捕捉长程依赖，而停用词往往不需要那么多计算资源。
- GQA 的局限：GQA 解决了 KV Cache 的带宽瓶颈，但 Query 侧的计算量依然随序列长度线性增长（整体复杂度仍受限于 O(N2)O(N^2)) 中的交互部分，且 Query 投影本身也是算力大户）。
作者的核心洞察是： 如果不同的 Token 需要不同的 Attention Head，那么为每个 Token 计算所有 Head 就是纯粹的算力浪费。 GQE 的目标是在保持 GQA 显存优势的同时，通过稀疏激活 Query Head 来进一步降低计算负载。
### 方法拆解：在 GQA 组内做“专家路由”
GQE 的设计非常克制且巧妙，它没有推翻 GQA，而是在其基础上叠加了一层 MoE 逻辑。
-组内路由（Within-Group Routing）
GQA 将 Query Head 分组，每组共享一个 KV Head。GQE 将每个组内的多个 Query Head 视为“专家池”。对于每个 Token，路由器只在当前组内选择 Top-k 个 Query 专家进行计算。
关键点：KV Head 始终保持 Dense（稠密）且不变。这意味着 GQA 带来的 KV Cache 显存节省完全保留，我们只节省了 Query 侧的计算和通信开销。
-混合输出构造：硬路由 + 软加权 + 共享锚点这是论文最核心的工程 Trick。单纯的 Top-k 选择是不可导的，路由器学不到东西。GQE 采用了三种输出的拼接：
Hard Concat：被选中的 k 个专家的直接输出（不可导部分）。
- Renormalized Weighted Sum：对被选中的专家输出进行加权求和，权重来自路由概率。这是唯一提供梯度给路由器的路径。
- Always-on Shared Head：一个始终激活的共享 Attention Head。它作为“锚点”，防止模型在训练初期因稀疏激活而崩溃。
⚠️ 反直觉发现 ：实验表明，如果没有那个“重新归一化的加权槽位”和“始终开启的共享 Head”，稀疏路由会导致精度大幅下降。这说明在注意力层做 MoE，稳定性比 MLP 层难得多。
### 关键结果：一半算力，同等精度作者在 2.5 亿参数、30B Token 预算下进行了严格对比。基线是全激活的 GQA（16 Query Heads / 8 KV Heads），GQE 变体每组只激活 1 个专家。

- 推理加速：在长上下文下，收益显著。

### 工程启示与局限对本地部署和 Agent 的意义：
对于运行长上下文 Agent 或本地大模型的用户，KV Cache 是显存杀手，但 Query 计算也是 GPU 算力的瓶颈。GQE 证明了在不增加 KV Cache 负担的前提下，可以通过稀疏化 Query 端获得近 2 倍的 Prefill 加速。这对于追求低延迟推理的场景极具吸引力。
局限与后续：
- 规模限制：目前仅在 250M 参数模型上验证。在更大规模模型（如 7B+）中，路由的复杂度和专家间的干扰可能不同，需要进一步验证。
- 输出维度变化：GQE 改变了 Attention 输出的维度（从 16 槽变为 10 槽），这意味着输出投影层 WOW_O​ 的形状需要调整，这增加了工程实现的复杂度，无法直接替换现有模型的权重。
总的来说，GQE 为“注意力层的条件计算”提供了一条可行的路径。它不是简单的剪枝，而是通过精心设计的梯度流和稳定锚点，让模型学会“按需分配”注意力资源。如果你正在优化长上下文推理性能，这个思路值得深入研究。
