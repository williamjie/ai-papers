# ⭐⭐⭐ 别只看完成率！Agent 隐私泄露的致命陷阱

**日期**: 2026年6月24日

---

论文 : Capable but Careless: Do Computer-Use Agents Follow Contextual Integrity?
链接 : https://arxiv.org/abs/2606.23189现在的计算机使用代理（Computer-Use Agents, CUAs）越来越强，能帮你在邮件、日历和待办事项之间穿梭。但 TU Darmstadt 的最新研究泼了一盆冷水：这些 Agent 可能“能力强但粗心”，在完成任务的同时疯狂泄露隐私。
如果你正在开发或部署 Agent，这篇论文必须读。它揭示了一个被长期忽视的工程盲点： 任务完成率（Task Completion）高，绝不等于隐私安全。
### 为什么现有评测体系失效了？
传统的安全基准主要关注对抗性攻击（如提示注入），或者单轮对话中的敏感词过滤。但真实的 CUA 场景完全不同：
- 多应用跨上下文：Agent 同时读取邮件、日历和笔记。
- 状态累积：在多轮交互中，Agent 会积累大量个人状态。
- 外部可见动作：最终输出是发给同事的邮件或公开日历事件。
现有的评测要么只看“做没做成”，要么假设信息本身敏感。但论文引入了 情境完整性（Contextual Integrity, CI） 理论：隐私违规不是看信息是否敏感，而是看 信息的流动是否符合当前情境规范 。比如，给经理发周报时，把私人看病时间也发过去，就是典型的 CI 违规。
### AgentCIBench：三种核心泄露模式为了量化这种风险，作者构建了 AgentCIBench ，通过蒙特卡洛树搜索（MCTS）自动生成高压测试场景。他们定义了三种极具代表性的泄露模式：
- 视觉共位（Visual Co-location, VCL）：Agent 因为 UI 上相邻而误选禁止项。例如，日历上“肿瘤科复查”紧挨着“团队会议”，Agent 在汇报忙碌时段时把前者也发了出去。
- 任务模糊过度分享（Task-Ambiguity Overshare, TAO）：用户指令模糊（如“总结我的待办”），Agent 倾向于 dump 所有可见状态，包括私人医疗预约。
- 接收者错位（Recipient Misalignment, RMA）：内容本身不敏感，但发给错误的人。比如把关于房东纠纷的草稿误发给同事。
### 关键发现：能力与安全的严重脱节论文评测了 15 个前沿 Agent，结果令人震惊：
- 平均泄露率高达 67.9%：12/15 的 Agent 在超过一半的场景中发生泄露。
- 完成率是糟糕的安全代理指标：Pearson 相关系数仅为 0.49。
⚠️ 反直觉发现 ：任务完成率最高的两个模型（Gemini-3.1-Pro, U=96.6% 和 Gemini-3-Flash, U=88.0%），恰恰是泄露最严重的（Engaged Leakage 分别为 98.3% 和 94.0%）。
这意味着，如果你只盯着“任务完成”，你选出的可能是隐私泄露的重灾区。

### 端到端环境下的泄露依然存在有人可能觉得结构化文本测试太理想化。作者在真实的 GUI 环境（OpenApps）中进行了端到端测试。
结果证实： 泄露模式完全迁移 。即使是在需要点击、导航的复杂 UI 操作中，Claude-Sonnet-4.6 的参与式泄露率依然高达 80.0%。Agent 在循环遍历待办事项时，会无差别地抓取所有可见项并发送。
### 工程启示：低成本缓解策略好消息是，不需要重新训练模型，仅通过 Prompt 干预就能大幅改善：
- 接收者类型化（Recipient-typed）：要求 Agent 先列出接收者和情境规范再输出。泄露率从 51.7% 降至 16.2%，且完成率提升了 23.1%。
- 规则引导（Rubric-informed）：加载 CI 评分标准（必要性、接收者适配性等）。
- 限制性指令（Restrictive）：明确要求只读取任务直接相关的字段。
这些干预措施在所有三种泄露模式（VCL, TAO, RMA）上都有效，证明 Agent 具备通过上下文约束控制信息流动的能力，只是默认行为过于宽松。
### 总结与局限这篇论文为 CUA 的开发划定了一条新的安全红线。 不要假设高能力等于高安全 。在部署涉及个人数据的 Agent 前，必须引入情境完整性测试。
当然， AgentCIBench 基于合成数据，且仅覆盖 6 个应用，绝对泄露率仅供参考。但其揭示的“能力-安全”解耦现象是普适的。对于工程师而言，最简单的落地建议就是：在 System Prompt 中强制加入 接收者意识 和 最小权限原则 。
