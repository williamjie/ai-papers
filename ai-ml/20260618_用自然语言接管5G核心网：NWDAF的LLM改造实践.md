# ⭐⭐⭐⭐ 用自然语言接管 5G 核心网：NWDAF 的 LLM 改造实践

**日期**: 2026-06-18

论文 : LLM-Enabled NWDAF: A Step Toward AI-Native 6G Network Intelligence链接 : https://arxiv.org/abs/2606.11877对于电信工程师而言，5G 核心网（5GC）的 Service-Based Architecture (SBA) 虽然灵活，但管理复杂度极高。NWDAF（Network Data Analytics Function）作为数据枢纽，原本需要编写复杂的 API 调用来获取指标或订阅事件。这篇论文展示了一个极具工程落地价值的尝试：通过集成大语言模型（LLM），让非专家也能用自然语言直接“对话”核心网，实现意图驱动的网络管理。
### 为什么现有方案不够用？
在传统的开源 5G 测试床（如 Free5GC）中，NWDAF 的功能往往局限于底层的指标暴露。操作员若想查询“当前活跃用户数”或订阅 AMF 的切换事件，必须熟悉 RESTful API 的结构、JSON 载荷格式以及具体的端点路径。
这种基于 CLI 或 API 的操作模式存在两个致命痛点：
- 门槛过高：网络运维人员通常不懂代码，而开发人员不熟悉 3GPP 协议细节。
- 缺乏语义抽象：操作员关心的是业务结果（如“监控区域 X 的注册失败”），而非具体的执行步骤。
随着 6G 向 AI-Native 演进，意图驱动网络（Intent-Based Networking, IBN）成为必然趋势。如何将人类的自然语言意图转化为标准的网络控制指令，是打通最后一公里的关键。
### 核心设计：RAG + 语义路由的极简架构论文没有选择微调一个巨大的垂直领域模型，而是采用了一种轻量级、可解释性强的 检索增强生成（Retrieval-Augmented Generation, RAG） 范式。其核心直觉在于： 将复杂的网络操作拆解为有限数量的标准意图类别，通过向量相似度匹配进行分类，再路由到对应的执行函数。
具体流程如下：
- 意图库构建：预定义七类操作意图，包括分析查询（Analytics Query）、AMF/SMF 订阅与取消订阅等。这些意图及其描述存储在本地数据库中。
- 语义嵌入与匹配：使用嵌入模型（Embedding Model）将用户输入的自然语言查询和预设意图库转换为高维向量。通过计算余弦相似度，系统能识别出即使用户措辞不同（如“Show me active users” vs “How many UEs are active?”），其核心意图也是一致的。
- 动态路由执行：
若分类为分析查询，系统自动生成 PromQL 语句并查询 Prometheus 获取实时指标。
- 若分类为控制指令，系统构造 JSON 载荷并通过 POST 请求调用 NWDAF 的 REST API 端点。
- 自然语言响应：LLM 接收原始数据（JSON 格式）和用户原始问题，生成人类可读的总结（如“当前有 52 个活跃 UE”）。
⚠️ 关键 Insight : 这种设计避免了 LLM 直接生成代码可能带来的幻觉风险。LLM 在此仅负责“理解意图”和“解释结果”，核心的网络交互逻辑由确定的规则引擎处理，兼顾了灵活性与安全性。
### 实验数据与性能表现作者在基于 Free5GC 和 UERANSIM 的开源测试床上进行了验证。为了模拟真实场景，他们实现了一个自定义的活动基移动性模型（Activity-Based Mobility Model），通过模拟 UE 在不同时间段（早、中、晚等）前往不同地点（家、工作、公园等）的行为，触发了大量的注册、去注册和切换事件。
关键实验结果如下：

### 工程启示与局限这篇论文对工程师最大的启发在于**“解耦”**。它展示了一种将 LLM 作为“中间件”而非“黑盒大脑”的架构模式。
- 低成本落地：无需昂贵的 GPU 集群进行大规模微调，现有的开源嵌入模型即可满足大部分意图识别需求。
- 标准化接口复用：通过映射到标准的 Prometheus 查询和 REST API，系统可以轻松扩展新的分析类型，只需在意图库中添加新条目并实现对应的后端函数即可。
然而，该方法也有明显边界：
- 意图覆盖有限：目前仅支持七类预定义意图，对于开放式的、未见过的新颖网络策略，系统无法处理。
- 缺乏复杂推理：LLM 仅用于生成回复和简单分类，并未利用其强大的逻辑推理能力进行复杂的故障根因分析或多步自动化编排。
尽管如此，作为一个开源的、可复现的测试床，它为研究 AI-Native 6G 网络管理提供了一个极佳的起点。代码和数据集的公开（GitHub: HenokDanielbfg/testbed）也极大地降低了后续研究的门槛。
