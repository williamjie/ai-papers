传统的 NL2Code（自然语言转代码）范式在处理视觉密集型任务时显得捉襟见肘。用文字描述一个复杂的 UI 布局或精确的几何结构，不仅效率低下，更极易丢失空间层级和样式细节。这篇来自美团、港大等机构联合发布的综述，系统梳理了 多模态代码智能（Multimodal Code Intelligence） 的最新进展，为工程师提供了一张从“看图说话”到“视觉驱动编程”的导航图。
### 为什么 NL2Code 不够用了？
现有方案的最大痛点在于 模态错位 。
在 Web 前端、数据图表或 CAD 设计中，代码的正确性不仅取决于语法，更取决于渲染后的布局、交互行为和领域约束。仅靠文本描述，模型很难对齐人类的高带宽视觉意图。多模态大语言模型（MLLMs）的引入，让模型能直接感知视觉输入，将视觉理解作为自动化编程的核心前置条件，而非辅助特征。
### 核心洞察：代码在视觉任务中的四种角色这篇综述最精彩的部分，不是罗列模型，而是重新定义了 代码在多模态任务中的角色 。作者指出，不同领域对代码的需求截然不同：
- 渲染产物（Rendered Artifact）：如 Web UI、SVG。重点是视觉保真度。
- 可编辑符号结构（Editable Symbolic Structure）：如科学文档、LaTeX。重点是结构完整性。
- 推理中间迹（Intermediate Reasoning Trace）：如程序化工具调用。代码是连接视觉感知与逻辑推导的桥梁。
- 可执行策略（Executable Policy）：如机器人控制、GUI 导航。代码是映射观察值到动作的结构化策略。
这种分类法直接指向了评估方法的差异：UI 生成看渲染相似度，而 Agent 任务看交互成功率。
### 关键结果：视觉相似 ≠ 功能正确论文揭示了一个反直觉的事实： 高视觉保真度往往掩盖了功能缺陷。
在 Web-to-Code 领域，早期基准主要依赖静态渲染评估。但随着模型能力提升，单纯追求“长得像”已无法保证代码可用。作者引用了 IWR-Bench 的实验数据：
- 模型的视觉保真度得分可达 64.25%- 但交互功能正确率仅为 24.39%⚠️ 警示 ：页面可以完美复刻参考图，却可能遗漏事件处理器、丢失状态转换或路由逻辑。这意味着基于静态截图的评估存在巨大盲区。
### 方法演进：从单模型到多智能体闭环针对上述痛点，工程实践正在经历三个阶段的演变：
- 直接监督微调（SFT）：如 WebSight、Web2Code，通过大规模合成数据训练基础基线。优点是简单高效，缺点是泛化能力弱，难以处理开放场景。
- 多智能体协作（Multi-Agent）：如 ScreenCoder、TDDev。将生成过程拆解为视觉定位、布局规划、代码实现和审查执行。这种模块化设计提升了可解释性，并允许在特定环节引入测试驱动开发（TDD）。
- 反馈驱动优化（Feedback-Driven）：如 WebGen-Agent、ReLook。引入闭环机制，利用浏览器渲染结果或计算机使用智能体（Computer-Use Agent, CUA）进行诊断和修复。CUA 通过实际执行任务来验证功能正确性，而非仅依赖视觉批评家。
### 工程启示与未来方向对于正在构建 AI 辅助编程工具的工程师，这篇论文提供了明确的落地指南：
- 评估体系升级：不要只盯着 SSIM 或 LPIPS 等视觉指标。必须引入基于执行的验证（Execution-based Verification），包括多状态测试和交互轨迹检查。
- 多信号验证（Multi-Signal Validation）：结合视觉证据、语义检查和代码级静态分析，避免单一信号的偏差。
- 可验证的 Agent 痕迹：未来的系统应能揭示 Agent 的动作是否真正 grounded 在视觉证据上，而非盲目猜测。
这篇综述不仅是一份文献清单，更是对多模态代码生成领域的一次深度体检。它提醒我们，从“模仿输出”走向“基于证据的可执行系统”，才是工程落地的终极目标。
