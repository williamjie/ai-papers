# ⭐⭐⭐½ LLM Agent 看代码仓库：视觉模态的坑与红利

**日期**: 2026-06-15

---

论文 : LLM Agents Can See Code Repositories链接 : https://arxiv.org/abs/2606.14061现有的 Coding Agent（如 SWE-Agent）处理代码库时，本质上是在做“文本压缩”。它们把复杂的文件夹层级、依赖关系全部拍平成线性 Token 序列。这导致模型在长上下文中容易迷失方向，反复执行 grep 和 cat 来拼凑结构信息，既慢又贵。
上海交通大学等机构的研究者提出了一个直觉上很性感的问题：既然人类开发者是“看”代码仓库的（通过 IDE 的文件树、依赖图），那多模态大模型（MLLM）能不能也直接“看”？他们构建了 SeeRepo 框架，首次系统性地验证了视觉模态在仓库级任务中的真实效用。
### 核心直觉：不要替换文本，要补充结构很多初涉多模态的工程师会想：“既然 MLLM 能看图，那我把代码渲染成图，是不是就能省掉昂贵的 Text Token？”
这是一个巨大的误区。 论文 RQ1 的实验给出了残酷的否定答案。当 Agent 被强制只能使用“纯视觉”交互（即 Bash 命令返回的是依赖图 PNG，而非文本）时，所有模型的性能都崩盘了：
- GPT-5-mini：Pass@1 从 55.0% 暴跌至 41.4%。
- Doubao-Seed-2.0-Lite：从 51.0% 惨跌至 16.9%。
- Kimi K2.5：从 70.3% 降至 55.0%。
更讽刺的是，成本不降反升。因为模型无法从图中提取精确的符号信息（如变量名、具体逻辑），它们会陷入“反复查询图形以补偿缺失细节”的死循环。Doubao 的成本甚至暴涨了 268% 。
核心 Insight 在于：视觉模态的优势不在于“承载代码内容”，而在于“提供全局拓扑导航”。 文本擅长精确的符号推理，而图像擅长展示空间关系和依赖流向。SeeRepo 的设计哲学是 Hybrid-Modality（混合模态） ：保留标准的文本接口用于读码和写码，仅引入视觉图作为“地图”辅助定位。
### 关键结果：省钱且准在 RQ2 中，研究者将 SeeRepo 生成的依赖图作为 补充上下文 提供给 Agent。结果令人振奋：视觉结构图帮助 Agent 更快锁定 Bug 位置，从而大幅减少了无意义的探索步骤。
以下是 SWE-bench Verified（500 个实例）上的关键对比数据：
模型 Pass@1 (准确率) Input Tokens (输入Token) Cost (成本) GPT-5-mini (Text) 55.0% 193,157 $0.031 GPT-5-mini (SeeRepo) 55.4% (+0.4%) 144,403 (-25%) $0.023 (-26%) GPT-5.1 (Text) 51.0% - $0.1795 GPT-5.1 (SeeRepo) 48.8% (-2.2%) - $0.0975 (-46%) Kimi K2.5 (Text) 68.8% - $0.1270 Kimi K2.5 (SeeRepo) 70.6% (+1.8%) - $0.1229 (-3%)
⚠️ 反直觉发现 ：最强的模型 GPT-5.1 准确率略有下降（-2.2%），但成本暴降 46%。这是因为强模型本身定位能力就强，视觉图让它“跑得太快”，偶尔会忽略一些细微的文本线索，但在工程上，用微小的精度换取近一半的成本节约是极具吸引力的 Trade-off。
### 工程启示：怎么用才最划算？
论文进一步拆解了视觉图的 布局 和 调用时机 ，给出了具体的落地建议：
-布局选择：Graph（图结构） 优于 Nested（嵌套）和 Tabular（表格）。
Graph 布局在保持准确率的同时，实现了最大的 Token 节省（-25% Input Tokens）。
- Tabular 虽然准确率最高（+1.2%），但成本节省较少。对于追求极致性价比的 Agent，Graph 是首选。
-调用时机：仅在“故障定位”阶段使用。
在 Localization（定位）阶段引入视觉图，Pass@1 提升 0.4%，成本降低 26%。
- 如果在 Repair（修复）或 Validation（验证）阶段才看，不仅没帮助，反而因为增加了上下文噪音导致准确率下降（Repair 阶段准确率跌了 5.0%）。
-动态深度：让 Agent 自己决定图的遍历深度（Hop Depth），比固定深度效果更好。Agent 能在简单问题时只看局部，复杂问题时展开全局，实现了效率与精度的最佳平衡。
### 局限与展望SeeRepo 目前主要依赖 AST 静态分析构建依赖图，这意味着它无法捕捉运行时动态行为。此外，视觉编码本身也有 Token 开销（ViT 将图像切分为 Patch），如果图形过于复杂，视觉 Token 的成本可能会抵消部分收益。
对于正在构建 Coding Agent 的工程师来说，这篇论文提供了一个明确的信号： 不要试图用图像替换代码文本 。相反，你可以为 Agent 增加一个“地图工具”，在探索阶段让它看一眼依赖关系图，这能显著减少它在文件海中盲目搜索的时间。在长上下文、高成本的现实场景中，这种“结构感知”能力比单纯的参数规模提升更具工程价值。
