论文 : FastContext: Training Efficient Repository Explorer for Coding Agents链接 : https://arxiv.org/abs/2606.14066让 LLM 修 Bug，最贵的往往不是写代码，而是找代码。Microsoft 这篇新工作直接点破了这个痛点：把“仓库探索”从主 Agent 中剥离出来，交给专门的子代理，不仅准了，还省了大量 Token。
### 痛点：探索是昂贵的噪音现有的 Coding Agent（如 Mini-SWE-Agent）通常让同一个模型既负责搜索又负责修复。这导致两个严重问题：
- Token 浪费：阅读和搜索占据了近 56% 的工具调用轮次，消耗了主 Agent 46.5% 的 Token 预算。
- 上下文污染：大量的无关代码片段堆积在历史对话中，干扰主 Agent 对核心逻辑的判断。
作者分析发现，在首次编辑前，Agent 平均需要 8.47 轮探索。未解决的案例往往陷入更长的无效搜索循环。这种“串行且嘈杂”的模式，是效率低下的根源。
### 方法：FastContext 子代理架构FastContext 的核心 Insight 很简单： 解耦 。它作为一个独立的只读子代理（Subagent），专门负责接收自然语言问题，通过并行工具调用（Read, Glob, Grep）定位代码，最终返回精简的文件路径和行号范围。
关键设计细节：
- 输出契约严格化：子代理不返回完整代码，只返回 <final_answer> 块中的文件路径和行号。这让主 Agent 能直接加载精准上下文，避免携带探索过程的噪音。
- 并行工具调用：支持在同一轮次中发出多个互补的搜索请求（如同时查符号、查路径），加速假设验证。
- 两阶段训练：
监督微调（SFT）：使用强模型（Sonnet 4.6）的轨迹数据，学习从宽泛搜索到精准引用的完整流程。
- 强化学习（RL）：基于任务奖励优化。奖励函数直接挂钩 Patch 衍生的文件/行 F1 值，并鼓励并行调用，惩罚格式错误。
⚠️ 反直觉发现 ：经过 RL 优化的 4B 参数小模型，在多个指标上竟然击败了未经 RL 优化的 30B 大模型。这说明针对特定任务（探索）的精准微调，比单纯堆砌参数量更有效。
### 实验结果：又快又准在 SWE-bench Multilingual、SWE-bench Pro 和 SWE-QA 三个基准测试中，FastContext 表现亮眼：
指标 GPT-5.4 (SWE-bench Pro) GPT-5.4 (SWE-QA) 解决率提升 +5.5% (46.0 -> 51.5) +0.7% Token 节省 -14.3% -60.3%- SWE-bench Pro：GPT-5.4 的解决率从 46.0% 提升至 51.5%，GLM-5.1 提升更显著（+5.0%）。
- Token 效率：在 SWE-QA 任务中，主 Agent 的 Token 消耗大幅降低 60.3%。这是因为子代理替主 Agent 承担了大部分阅读和搜索工作。
- 定位精度：在独立探索质量评估中，FastContext-4B-RL 的文件级 F1 达到 71.48，优于其他非前沿模型的基线（如 CodeScout-14B 的 68.57）。
### 工程启示- 模块化 Agent 设计：不要试图让一个通用模型包揽所有工作。将“探索”、“推理”、“执行”拆分为不同能力的子代理，能显著提升系统上限。
- 小模型的大作用：4B 参数的模型经过针对性 RL 训练后，完全可以胜任复杂的仓库导航任务。这在成本敏感的部署场景中极具价值。
- RL 的价值：SFT 只能模仿行为，RL 才能对齐目标。在探索任务中，直接优化“找到正确代码位置”的奖励，比单纯模仿人类搜索轨迹更有效。
### 局限与展望目前 FastContext 仅集成于 Mini-SWE-Agent，且主要验证了强主模型（GPT-5.4, GLM-5.1）的效果。未来需要探索其在更广泛 Agent 框架中的兼容性，以及配合更小参数主模型（如 30B 以下）的表现。此外，如何进一步压缩子代理至 1.7B 甚至更小，也是值得关注的方向。
