论文 : FVSpec: Real-World Property-Based Tests as Lean Challenges链接 : https://arxiv.org/abs/2606.01008如果 AI 生成的代码要真正进入生产环境，形式化验证（Formal Verification, FV）就是那道绕不过去的坎。但现有的基准测试要么太学术（数学家写的定理），要么太竞技（程序员写的算法题）。这篇论文直接干了一件大事：从 GitHub 上扒了 1.1 万个真实的属性测试（Property-Based Tests, PBT），把它们变成 Lean 形式化验证的挑战题。
### 为什么现有的 Benchmark 不够用？
目前的 ITP（交互式定理证明器）基准测试存在严重的“分布偏移”。
看看 Table 1，像 Minif2f、PutnamBench 这些主流数据集，全是数学奥林匹克或大学数学题。即使是面向软件的 Vericoding 或 LEAN-GitHub，代码也是专家专门为验证写的，或者是从竞赛中抓取的。
现实中的软件工程完全不同：
- 黑盒依赖：代码充斥着 Redis、HTTP 请求、文件系统调用。
- 非确定性：并发、时间戳、外部 API 让传统数学证明难以直接套用。
- 工程复杂度：普通的 Python 项目依赖树庞大，逻辑分散，根本不是单体封闭系统。
如果你只用数学题训练 AI，它可能连一个带副作用的生产函数都验证不了。FVSpec 的核心动机就是填补这个“现实代码”与“形式化证明”之间的鸿沟。
### 核心设计：从 PBT 到 Lean 的自动转译论文没有让人类专家去手写证明，而是构建了一个自动化流水线，将 Python 的 Hypothesis 测试自动转化为 Lean 4 的定理。
1. 数据源清洗（FVSpec:PBT）
作者爬取了 GitHub 上依赖 Hypothesis 的仓库，经过去重和许可证过滤，得到了 11,039 个真实的 PBT。这些数据来自 333 个不同的开源项目，涵盖了从日期处理到 ML 框架的各种场景，非常“野”。
2. 智能转译管线（Agentic Transpilation）
这是最巧妙的部分。PBT 是统计性的（随机测试），而 FV 是确定性的（数学证明）。如何转换？
- 函数发现：使用 tree-sitter 提取被测函数及其本地依赖。
- LLM 代理转译：让 LLM 将 Python 代码和测试断言翻译成 Lean 的 Impl.lean（实现）和 Spec.lean（规格说明）。
- LSP 自修复循环：翻译往往第一次就会报错。系统利用 Lean 语言服务器协议（LSP）获取编译错误，反馈给 LLM 进行最多 16 轮迭代修复，直到代码能通过 lake build。
⚠️ 关键 Insight ：对于涉及副作用的代码（如 Redis 操作），LLM 不会尝试模拟 Redis 内部逻辑，而是将其抽象为“未解释公理”（Uninterpreted Axioms）。这意味着证明的是状态转换的逻辑正确性，而非具体底层实现。这既保留了验证价值，又避免了不可计算的复杂性。
### 实验结果：AI 真的能证吗？
最终产出了 9,415 个 Lean 验证挑战（平均每个 PBT 生成约 3 个形式化版本）。作者用 Claude Sonnet 4.6、Opus 4.7 和 GPT 5.4 进行了基线测试。

- Hard 题目很难：即使是 Opus 4.7，在 Hard 难度上也只有不到 60% 的通过率。这说明形式化验证对当前 LLM 依然是硬骨头。
- 拒绝率差异：GPT-5.4 在 Easy 题目上有 25% 的拒绝率（认为不可解），而 Opus 和 Sonnet 仅约 4%-5%。这可能反映了不同模型对“可证明性”的判断阈值不同。
- 计算成本高昂：Figure 8 显示，每个样本的平均 token 消耗高达数千甚至上万，且耗时较长。这不是一个能低成本大规模微调的任务。
### 工程启示- PBT 是 FV 的最佳跳板：很多团队已经在用 Hypothesis 做测试。FVSpec 证明，这些现成的测试用例可以直接转化为形式化验证的语料库，无需从零开始。
- LSP 反馈闭环至关重要：单纯让 LLM “写代码”效果有限。引入编译器错误作为反馈信号（ReAct 模式），能显著提升生成代码的可编译性。这在任何代码生成场景中都是通用最佳实践。
- 抽象副作用是必经之路：想要验证真实业务逻辑，必须接受“部分黑盒化”。不要试图证明 Redis 的内存分配算法，而是证明“调用 Redis 后状态符合预期”。
### 局限与展望论文坦诚地指出了几个边界：
- 忠实度问题：自动转译并非完美等价。结构忠实度中位数仅为 0.65，参数覆盖和依赖覆盖是短板。
- 公理滥用风险：LLM 可能会引入额外的强公理（如选择公理）来“作弊”通过编译，虽然论文记录了这些依赖，但自动检测仍需改进。
- 缺乏人类基线：我们不知道人类专家在这些题目上的表现，因此难以准确评估 AI 的相对水平。
FVSpec 为 AI 辅助形式化验证提供了一个稀缺的、面向真实工程的数据集。它提醒我们：未来的安全关键软件，其验证工作流很可能就是“PBT -> 自动转译 -> LLM 证明 -> 人工审计”。
