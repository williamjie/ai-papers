# ⭐⭐⭐⭐ 4B模型吊打671B：Pythagoras-Prover的 Lean 证明新范式

**日期**: 2026-06-16

---

论文 : Pythagoras-Prover: Advancing Efficient Formal Proving via Augmented Lean Formalisation链接 : https://arxiv.org/abs/2606.12594当前形式化证明（Formal Proving）领域存在一个尴尬的共识：想要高性能，就得堆算力。
主流方案要么依赖数百亿参数的巨型模型，要么在推理阶段进行昂贵的搜索与自我修正。这种“暴力美学”让大多数工程师望而却步，因为高昂的训练和推理成本直接劝退了本地部署的可能性。
这篇论文带来的冲击在于，它用极小的参数规模实现了超越巨头的性能，并给出了一套可复现的高效数据构建流水线。
问题与动机：算力焦虑与数据瓶颈现有的 Lean 定理证明器面临两大痛点：
- 验证数据稀缺且昂贵：Lean 的类型检查器虽然能保证正确性，但吞吐量极低。每一行代码的验证都伴随着巨大的时间成本，导致高质量训练数据难以规模化生产。
- 推理链条过长：形式化证明需要极长的上下文窗口。现有的监督微调（Supervised Fine-Tuning, SFT）往往受限于上下文预算，强行截断会导致逻辑断裂，性能大幅跳水。
传统的解决思路是“大力出奇迹”，但 Pythagoras-Prover 团队认为，核心瓶颈不在模型规模，而在 数据构建策略 和 训练效率 。
方法拆解：从“暴力验证”到“智能增强”
论文的核心洞察在于：我们不需要验证每一个变体，只需要保证核心逻辑的一致性。为此，他们提出了 Augmented Lean Formalisation (ALF) 技术。
ALF 的设计直觉非常巧妙：
- 结构化变异：不直接生成新题目，而是对已有的已验证（Verified）种子数据进行五种结构化突变：简化、泛化、引理提议、步骤分解和重构。
- 轻量级一致性检查：这是最关键的一步。传统方法要求每个变体都经过 Lean 编译器验证，耗时极长。ALF 仅使用一个“陈述-证明对齐”检查（Statement-alignment check）。只要生成的证明引用了目标陈述，就保留该样本。
- 自蒸馏放大：通过这种低成本过滤，他们将种子语料库扩大了约 2.5 倍，且无需额外的验证算力。
在训练策略上，他们采用了 课程学习（Curriculum Learning） 与 动态推理过滤 ：
- 难度分层：将数据分为 Easy、Medium、Hard 三个层级，模型从短证明逐步过渡到长证明，避免初期陷入复杂逻辑的泥潭。
- 8K 上下文预算下的智能截断：在 8K token 的限制下，如果“推理链+证明”超长，则丢弃推理链保留证明；如果仅证明就超长，则直接丢弃。这种策略在有限资源下最大化了有效信息的摄入。
此外，论文还尝试了基于扩散模型（Diffusion-based）的证明生成。通过 Tactic-based Masking ，以完整的 Lean 策略（Tactic）为单位进行掩码和去噪，而非单个 token。这不仅符合数学推理的粒度，还使得推理速度比自回归模型快 2.58 倍。
关键结果：小模型的逆袭实验数据证明了这套“算力吝啬”方案的有效性。在 MiniF2F-Test 基准上：

工程启示：如何落地形式化证明？
对于希望将形式化证明引入 Agent 或本地工具的工程师，这篇论文提供了三条明确路径：
- 数据质量 > 模型规模：4B 模型能打败 671B 模型，核心在于 ALF 带来的高质量、结构化变体数据。在资源有限时，优先优化数据流水线而非盲目扩参。
- 放弃全量验证的执念：ALF 证明了“一致性检查”足以替代部分“编译器验证”。在构建合成数据时，可以利用轻量级 LLM 进行逻辑对齐，大幅降低数据生产成本。
- 扩散模型的潜力：虽然目前 Diffusion 模型在准确率上略逊于自回归模型，但其推理速度优势明显（2.58x）。对于对延迟敏感的场景，基于 Tactic 的扩散模型是一个值得探索的方向。
局限与展望尽管性能出色，但 ALF 生成的变体并未全部经过 Lean 验证。论文随机抽样显示 87.8% 的通过率，意味着仍有约 12% 的数据存在潜在错误。此外，扩散模型目前仅作为概念验证（Proof-of-Concept），在复杂长程推理上的稳定性仍需进一步打磨。
总的来说，Pythagoras-Prover 证明了形式化证明不必是“富人的游戏”。通过聪明的数据工程，小模型也能在严谨的数学世界中游刃有余。
