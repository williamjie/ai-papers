论文 : RedAct: Redacting Agent Capability Traces for Procedural Skill Protection链接 : https://arxiv.org/abs/2606.10813Agent 的轨迹（Trajectory）不仅是调试工具，更是核心资产。这篇论文揭示了一个被忽视的风险：公开的执行日志会泄露私有技能（Procedural Skills），而 RedAct 提供了一套既能保护隐私又能保留审计证据的工程方案。
### 痛点：公开日志 = 免费开源技能包随着 Agent 从原型走向生产，企业越来越依赖包含专有工作流、校准阈值和错误恢复逻辑的“技能包”。为了透明度和可调试性，这些执行轨迹常被公开或共享。
然而，下游攻击者无需访问模型权重，仅通过分析这些轨迹，就能重构出关键的业务逻辑。这构成了**黑盒轨迹泄露（Black-Box Trace Disclosure）**问题：
- 风险本质：不是简单的答案抄袭，而是可复用程序知识（如公式、API 选择序列）的提取。
- 现有方案失效：仅保留最终答案会丧失审计价值；通用摘要无法可靠分离执行证据与受保护的程序细节。
### 方法拆解：RedAct 的双层防御直觉RedAct 的核心 Insight 在于： 脱敏不是简单的模糊化，而是基于关键信息的精准重写。 它假设技能所有者拥有受保护的技能包，从而能精准定位哪些信息必须隐藏。
框架分为两个互补层：
-关键信息引导的重写层（Rewriting Layer）
定位：利用 LLM 从任务指令和技能包中提取受保护的关键项（KprotK_{prot}​），如特定公式、阈值或专用工具选择。
- 重写：将暴露这些关键项的中间步骤抽象为高层描述，同时保留最终输出、工具调用证据和执行顺序。
- 设计直觉：通过显式定位受保护项，避免通用重写中残留的可复用信号。
-行为水印层（Behavioral Watermarking Layer）
注入：在轨迹中插入功能中立的行为钩子（Hooks），如固定的环境检查或错误恢复短语。
- 溯源：当怀疑下游模型窃取了轨迹时，通过激活词触发检测，验证其是否复现了这些特定行为。
- 设计直觉：水印不干扰任务语义，但提供了统计意义上的所有权证据。
### 关键结果：从“高价值泄露”到“低于基线”
论文构建了 CAPTRACE BENCH ，包含 75 个长_horizon_任务和 154 个技能。实验对比了原始轨迹（Raw Trace）与 RedAct 保护后的轨迹在多种复用方法下的表现。
1. 原始轨迹是巨大的攻击面在无技能基线（No Skills）下，平均成功率（SR）为 45.2%。使用原始轨迹进行技能提取后，平均步骤成功率（SSR）飙升至 73.5% ，几乎追平了拥有完整技能包（Oracle Skills, 76.5%）的效果。
- 归一化技能转移（NST）：原始轨迹的 NST 高达 44.7–67.1%，证明大量程序知识被泄露。
2. RedAct 有效切断技能转移经过 RedAct 处理后，所有复用方法的 SSR 均降至 68.0% 以下 ，低于无技能基线。
- NST 归零甚至为负：保护后的 NST 降至 -5.9%，意味着下游模型从轨迹中获得的程序优势完全消失。
- 关键信息恢复率（RPI）：下降 37–48%，表明残留的可复用内容大幅减少。
3. 水印溯源高准确率在 Qwen3-8B 学生模型上测试行为水印：
- Env Check 水印：真实检测率（TD）达 93.6%，误报率（FA）仅 1.3%。
- Ritual Marker 水印：TD 接近 100%，FA 为 0.0%。

对于正在构建企业级 Agent 的团队，这篇论文提供了明确的行动指南：
- 不要直接公开原始轨迹：尤其是包含复杂工作流、错误恢复逻辑和专用工具调用的部分。这些是攻击者进行技能蒸馏（Skill Distillation）的金矿。
- 实施选择性脱敏：使用类似 RedAct 的方法，先识别“关键程序项”，再进行抽象重写。保留最终结果和工具名称以满足审计需求，但隐藏具体的参数配置和中间决策逻辑。
- 嵌入行为水印：如果必须公开轨迹用于社区分享或调试，注入无害的行为钩子。这不仅能威慑潜在窃取者，还能在发生泄露时提供取证依据。
### 局限与展望RedAct 目前主要验证于受控的长_horizon_任务环境。实际部署中，噪声更大的轨迹和动态变化的工具环境可能影响脱敏效果。此外，上下文相关的水印（Contextual Hooks）虽然误报率低，但检测率也较低（16.4–32.2%），未来需要更强健的触发机制。
总之，Agent 轨迹应被视为 安全接口 。在追求透明度的同时，必须通过工程手段剥离其中的可��用程序资产。
