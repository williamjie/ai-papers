# ⭐⭐⭐½ LLM压缩别乱搞：输出口省钱，输入口亏钱

**日期**: 2026-06-25

---

论文 : CAVEWOMAN: How Large Language Models Behave Under Linguistic Input and Output Compression链接 : https://arxiv.org/abs/2606.24083为了省 Token 成本，很多工程师喜欢搞“原始人风格”（Caveman Style）：砍掉介词、连词，只留核心名词动词。这篇论文直接泼冷水： 压缩输入和压缩输出，结果截然相反。 盲目压缩 Prompt 不仅不省钱，反而会让模型啰嗦，导致总成本飙升。
### 痛点：只看准确率是“盲人摸象”
过去大家评估压缩效果，只看两件事：Token 少了没？任务准确率掉了没？
这有个巨大的逻辑漏洞： 输入 Token 便宜，输出 Token 贵（通常 4-8 倍）。
如果你为了省 10 个输入 Token，导致模型多输出了 50 个 Token 来“补全”信息，那你其实是亏大了。现有研究把这两个渠道分开看，且只盯着准确率，完全忽略了真实的经济账。
### 方法拆解：CAVEWOMAN 双通道评估作者提出了 CAVEWOMAN 协议，核心直觉是 控制变量 。
他们设计了 5 个语言压缩等级（L0-L4），从保留全部语法到只剩名词骨架。关键设计在于两个独立通道：
- 输入压缩（Condition A）：把 Prompt 里的功能词（如 the, a, is）删掉，看模型反应。
- 输出压缩（Condition B）：Prompt 不变，但在 System Prompt 里强制要求模型用“电报体”回答。
这种方法能剥离出“模型到底需不需要完整的语法提示”以及“模型是否能在受限表达下保持逻辑”。
### 关键结果：反直觉的成本不对称实验覆盖了 8 个主流模型（GPT-4o, Claude Sonnet, Qwen 等）和 5 个数据集。数据非常硬核：
1. 输出压缩是正收益，输入压缩是负收益- 输出压缩：在大多数 API 模型上，单次请求成本降低 1.4–2.4 倍，最佳情况甚至达到 3 倍。因为输出 Token 单价高，压住它直接省钱。
- 输入压缩：这是个“双输”局面。虽然 Prompt 变短了，但模型为了理解残缺的指令，倾向于生成更长的解释性回复。净成本反而上升 1.15 倍（平均），在特定数据集上甚至高达 2.7 倍。
2. 准确率与文本一致性解耦这是最吓人的发现。在非推理类模型上， 51.9% 的“正确”回答，其表面文本已经无法通过双向 NLI（自然语言推理）判定为与原始完整回答一致。
⚠️ 警示 ：这意味着模型虽然答对了题，但它的“思考路径”或“表述逻辑”已经偏离了正常状态。如果你依赖模型的中间过程做审计或 Agent 路由，这种压缩会埋雷。
### 工程启示：怎么压才不亏？
- 优先压输出，慎压输入：除非你的 Prompt 是几万字长文且模型有强大的上下文窗口管理能力，否则不要试图通过删除功能词来优化短 Prompt。这只会触发模型的“补偿性冗长”，增加成本并降低稳定性。
- 根据任务类型选择策略：对于 BoolQ、多选题等答案简短的任务，输出压缩收益最大（直接砍掉废话）。对于 GSM8K 这种需要多步数学推理的任务，输出压缩能保留准确率的同时大幅削减 Token，是性价比最高的手段。
- 模型鲁棒性差异巨大：Gemma-4-E4B 在输出约束下表现最稳（准确率比率 3.7x），而 GPT-5.4 反而最脆弱。选型时不能只看基准测试的绝对分数，要看它在“受限表达”下的相对表现。
### 局限与展望论文也承认，目前的评估主要基于短答案任务（数学、多选）。对于长文本生成（如摘要、创作），这种“电报体”约束可能导致语义丢失更严重。此外，双向 NLI 只能检测表面文本差异，深层命题漂移仍需更多探索。
总之，别再把 Prompt 当垃圾清理了。想省钱，去管住模型的嘴，而不是用户的输入。
