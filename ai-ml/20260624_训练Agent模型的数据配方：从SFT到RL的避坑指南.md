# ⭐⭐⭐½ 训练 Agent 模型的数据配方：从 SFT 到 RL 的避坑指南

**日期**: 2026年6月24日

论文 : OpenThoughts-Agent: Data Recipes for Agentic Models链接 : https://arxiv.org/abs/2606.24855做 AI Agent 的朋友都知道，现在的瓶颈往往不在模型架构，而在数据。这篇来自 Berkeley、Stanford 等机构的合作论文，直接撕开了“黑盒”训练的一角，公开了一套完整的 Agent 数据清洗流水线（Pipeline）。
它没有搞花哨的新算法，而是通过超过 100 次控制变量实验，把 SFT（Supervised Fine-Tuning, 监督微调）和 RL（Reinforcement Learning, 强化学习）的数据来源、筛选策略扒了个底朝天。对于想自己微调 Agent 模型的工程师来说，这简直是一份实打实的“避坑指南”。
### 现有方案的痛点目前的开源 Agent 数据项目（如 SWE-Smith, Nemotron-Terminal）大多只盯着单一基准测试优化。比如有的专攻代码修复，有的专攻终端操作。这导致模型在特定任务上很强，但泛化能力极差。
OpenThoughts-Agent 的目标很明确：构建一个能在多个异构 Agent 任务（软件工程、终端操作、金融、医疗等）上通用的高性能数据配方。他们基于 Qwen3-32B 进行了微调，并在 7 个基准测试中取得了平均 44.8% 的准确率，比最强的开源竞品 Nemotron-Terminal-32B 高出 3.9 个百分点。
### 核心 Insight：SFT 数据流水线的反直觉发现论文最值钱的部分在于对 SFT 数据流水线六个阶段的消融实验。这里有几个完全反常识的发现，直接颠覆了我们的惯性思维。
1. “最强模型”不等于“最好老师”
直觉告诉我们，用最强的模型生成轨迹（Trajectory），学生模型肯定学得最好。但实验结果啪啪打脸。
⚠️ 反直觉发现 : 尽管 GPT-5.3-Codex 在基准测试中表现最强，但它作为“老师”生成的数据，训练出的模型在 Terminal-Bench 2.0 上反而比用较旧的 GLM-4.7-AWQ 生成的数据差了约 5%。
这说明，最强的推理能力并不等同于最适宜教学的轨迹风格。GLM-4.7-AWQ 可能提供了更清晰、更易于模仿的中间步骤，而最强模型可能跳步太多或引入了难以复现的复杂逻辑。
2. 任务多样性 > 单一来源的深度挖掘在数据混合策略上，单纯堆砌排名最高的单一数据源（Top-1）会导致过拟合。论文发现，混合 Top-4 到 Top-8 的数据源效果最佳。
更重要的是，当数据规模扩大到 100K 时，简单重复同一批任务描述（Upsampling）会导致性能 plateau（停滞）。真正的增益来自于 合成增强（Synthetic Augmentation） ：通过对原始问题进行指令重写，在不引入新底层问题的前提下，将表面形式（Surface Forms）从 ~900 扩展到 21K+。这证明了“任务描述的多样性”是突破性能瓶颈的关键。
3. 轨迹长度筛选的硬指标在过滤 Agent 执行轨迹时，简单的启发式规则非常有效。论文发现，保留至少 5 个回合（Turns） 的轨迹能显著提升下游表现。
这背后的逻辑很朴素：短轨迹往往意味着任务太简单或模型过早放弃，缺乏足够的多轮推理监督信号。只要确保训练数据包含足够长的交互过程，模型就能学到更稳健的工具使用习惯。

### 工程启示- 别迷信最强 Teacher：如果你资源有限，不要用最强的闭源模型生成所有数据。尝试一些中等规模、风格稳健的开源模型（如 GLM 系列）作为 Teacher，可能获得更好的 SFT 效果。
- 重视轨迹长度：在清洗 Agent 日志时，直接丢弃少于 3-5 轮交互的样本。这些“快餐式”回答对提升复杂任务能力帮助甚微。
- 合成增强是扩量的正道：当高质量原始数据耗尽时，不要盲目增加重复采样。利用 LLM 对现有问题进行指令重写（Paraphrasing），保持底层逻辑不变但改变表面表述，能有效提升模型的泛化能力。
- RL 数据源的选择至关重要：RL 阶段的数据来源差异巨大。论文发现，经过重构的竞赛编程问题（pymethods2test）在 RL 中表现最佳，因为它提供了清晰、可验证的单函数修正信号，避免了真实仓库中复杂的依赖和环境噪声。
### 局限与展望目前该研究主要在 8B 和 32B 规模上验证，RL 部分仅在 8B 模型上进行。对于更大参数量的模型（如 70B+），同样的数据配方是否依然有效，仍需进一步探索。此外，所有实验均基于 Qwen3 基座模型，不同基座模型的适配性尚未完全解耦。
总的来说，这篇论文的价值不在于提出了什么新算法，而在于它用严谨的实验数据，回答了“如何构建 Agent 训练数据”这个工程界最头疼的问题。代码和数据集已开源在 openthoughts.ai，建议直接 clone 下来跑一遍流水线，体会其中的细节差异。

挑刺:
-遗漏关键负结果：任务描述增强策略无效。 博文在“核心 Insight”中只列举了三个正向发现，但原文 Section 3.3 表 4 明确显示所有 LLM 驱动的任务增强（Harden、Constrain 等）均未能可靠超越“不做任何增强”的基线，且“Original (no augmentation)”排名第一。博文完全没有提及这一重要负结果，却随后在“工程启示”中建议“合成增强是扩量的正道”，可能让读者误以为所有形式的增强都有益。实际上原文的合成增强特指对 Tezos 问题进行指令重写以增加表面形式多样性，而非直接增强任务描述本身。
-评估设置关键条件缺失。 博文在“关键结果对比”表格中直接列出 OT-Agent-32B 与 Nemotron-Terminal-32B 的数值，但未说明原文中每个模型的成绩是“在两个评估 harness（Terminus-2 和模型原始 harness）中取最大值”（原文 Section 3 Evaluation Setup 及表 1 脚注）。这一约束对理解数字可比性至关重要，缺失可能导致读者误认为这些是单一标准下的直接对比。
-教师模型推荐过度简化。 博文在“工程启示”中建议“尝试一些中等规模、风格稳健的开源模型（如 GLM 系列）作为 Teacher”。但原文表 6 中最佳教师 GLM 4.7 是 AWQ 量化版本（GLM-4.7-AWQ），而 GLM 5（非量化）排名第三，且量化这一细节可能影响可复现性。博文未提及量化因素，也未指出原文并未解释量化为何可能有助于教学，这种简化可能误导读者在选择教师时忽略量化与否的关键差异。
