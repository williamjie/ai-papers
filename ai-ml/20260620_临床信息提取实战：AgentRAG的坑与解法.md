论文 : Configurable Clinical Information Extraction with Agentic RAG: What Works, What Breaks, and Why链接 : https://arxiv.org/abs/2606.19602大多数医疗 NLP 论文还在跑 benchmark，这篇直接部署在德国埃森大学医院，面对近 20 亿 FHIR 资源的真实数据。它揭示了一个残酷真相：临床数据的元数据质量远不足以支撑传统 RAG，必须依靠 Agent 进行内容级推理。
### 为什么传统 RAG 在临床场景会崩？
临床信息提取（Clinical Information Extraction, IE）长期依赖开发者硬编码规则或针对特定任务微调模型。LLM 虽然能零样本提取，但面临两大落地障碍：隐私合规要求本地部署，以及真实病历的复杂性。
标准检索增强生成（Retrieval-Augmented Generation, RAG）依赖元数据进行过滤和排序。但在该医院的 FHIR 仓库中，文档级元数据极度稀疏且不可靠：
- 时间戳错乱：56.5% 的文档时间戳与其所属就诊时段不符，仅 58.8% 与文档内容提取的日期一致。
- 结构缺失：99.5% 的文档缺乏关系标识，作者信息缺失率高达 98.1%。
- 去重难题：每个患者平均有 33.5% 的重复文档，最高达 54.6%。
这意味着基于元数据的静态过滤（如按日期、科室筛选）完全失效。系统无法知道哪份报告是最新的，也无法通过标题判断内容相关性。
### 核心设计：用 Agent 弥补数据缺陷作者提出的 ACIE 系统核心洞察是： 当数据质量不足以支持检索时，推理必须介入检索过程。
- Agent 驱动检索：不再预先过滤，而是让 Agent 像医生一样“阅读”。它通过工具调用语义搜索全文、生成文档摘要预览、检查具体内容。这种迭代式决策解决了元数据缺失导致的定位难题。
- 长度惩罚机制：临床文档包含大量表格和表单，分块后产生许多短片段。密集检索器倾向于偏好这些短片段（Short-chunk bias）。作者引入长度惩罚公式 s=sim(q,c)⋅p(ℓ)s = sim(q, c) \cdot p(\ell)sim(q,c)⋅p(ℓ)，抑制短碎片得分，确保上下文完整性。
- Markdown 优于 JSON：这是一个反直觉的工程发现。在输入侧，将患者元数据从 JSON 转换为 Markdown 格式，消除了特定患者触发的确定性工具调用错误。复杂的嵌套 JSON 结构会干扰 LLM 的解析逻辑。

### 工程启示与局限- 数据决定架构：不要试图用完美的 RAG 管道去适配糟糕的数据。当元数据不可信时，必须增加 Agent 的推理层来动态判断文档价值。
- 本地部署可行性：使用 Qwen3.6-35B-A3B 混合专家模型在 4x H100 上运行，证明了高性能本地临床 AI 的技术路径。
- 人机协作新模式：系统并非替代医生，而是将角色从“数据录入”转变为“验证者”。医生平均每人节省约 3 倍的时间。
局限在于评估仅针对单一疾病（淋巴瘤）和单中心数据。日期和表格处理的低准确率表明，跨文档的时间逻辑推理仍是当前 Agent 的短板。对于需要极高时间一致性的场景，仍需更复杂的时序建模或人工深度介入。
