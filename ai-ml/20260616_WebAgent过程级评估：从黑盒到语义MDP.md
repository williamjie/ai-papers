论文 : Where Did It Go Wrong? Process-Level Evaluation of Web Agents with Semantic State Tracking链接 : https://arxiv.org/abs/2606.15673做 Web Agent 的朋友都知道，现在的评测太“粗糙”了。要么成功，要么失败，中间过程全被黑盒吞掉。这篇论文提出了一个很痛的点： 终端成功率（Terminal Success）掩盖了模型能力的巨大差异 。它引入了 WEB STEP 基准，通过语义状态追踪，把 Agent 的每一步操作都拆解得明明白白。
## 痛点：为什么只看结果不够？
现有的 Web Agent 评测（如 WebArena, Mind2Web）大多只关注最终是否完成任务。但这就像考卷只给总分不给错题分析。
想象两个 Agent 在找消息并点赞：
- Agent A：找到了正确的对话窗口，但点错了表情符号。
- Agent B：根本没找到目标窗口，瞎点了一通后超时。
在传统评测里，两者得分都是 0。但对工程师来说，Agent A 只是“执行”出了点小错，Agent B 则是“探索”能力完全失效。 缺乏过程信息，我们就无法进行准确的信用分配（Credit Assignment），也就不知道该怎么优化模型。
## 核心设计：语义 MDP 双生体论文的核心 Insight 是构建一个**语义马尔可夫决策过程（Semantic MDP）**作为网页的“双生体”。
- GUI 层：Agent 依然像普通人一样，通过点击坐标、输入文本与渲染后的界面交互。
- 语义层：后台有一个确定性的状态机。每当 Agent 在 GUI 上操作，系统会自动将其映射为高层语义动作（如 Search, Filter, Inspect）。
这种设计的精妙之处在于： 它不需要人工标注轨迹 。因为网页本身就是由这个 MDP 驱动渲染的，所以每一个 GUI 操作背后对应的语义状态转移是精确可恢复的。这使得我们能在无需额外成本的情况下，自动记录 Agent 到底“看到了什么”、“做了什么决策”。
## 关键发现：隐藏的行为差异论文在 WEB STEP（1800 个任务实例）上测试了 5 个主流模型。结果非常有意思， 过程指标揭示了终端指标看不见的真相 。
### 1. 小模型的“性格”迥异看下面这张表，三个小模型的成功率几乎一样（31%-33%），但行为模式完全不同：
模型 终端成功率 探索成功率 执行成功率 步数效率 UI-TARS-7B 32.6% 46.3% (最高) 50.6% (最低) 低 (步数多) GUI-Owl-8B 31.9% 43.6% 55.6% 中 Fara-7B 31.4% 43.6% 55.7% 高 (步数少)
- UI-TARS 是个“探索狂魔”：它经常能找到目标（探索率高），但在最后一步搞砸了（执行率低），且步数冗余。
- Fara 则比较“急躁”：步数最少，覆盖信息最少，说明它还没看清就尝试提交。
⚠️ 反直觉发现 ：在 Housing 数据集上，OpenAI CUA 比 Qwen3.5 在 Commit （最终提交）技能上强 23.7% ，但在 Filter （筛选）技能上却弱 15.6% 。如果不拆解到技能级别，你会误以为 CUA 在这个网站全面领先或落后，从而给出错误的优化方向。
### 2. 失败点的“分叉口”分析论文通过对比成功和失败轨迹在语义状态上的首次分歧点（Bifurcation），定位了致命错误：
- GUI-Owl 最容易在 Filter 环节走偏（24% 的错误分支源于此）。
- Qwen3.5 和 OpenAI CUA 更多是在 Inspect（查看详情）时出错，即点开了错误的商品详情页。
- GUI-Owl 还有一个独特问题：65% 的“延迟提交”失败是因为它在应该结束时还在反复 Filter，而 Qwen3.5 则倾向于继续 Navigate 或 Inspect。
### 3. 难度越高，差异越大随着任务中“硬负样本”（Hard Negatives，即看起来像目标但实际不是的干扰项）数量增加，模型间的差距被拉大。OpenAI CUA 对干扰项数量的敏感度远低于其他模型，显示出更强的探索鲁棒性。
## 工程启示这篇论文给 Agent 开发者的建议非常具体：
- 不要只盯着 Success Rate：如果你的模型成功率卡在瓶颈期，去分析它的探索覆盖率和步数效率。是找不着北（探索差），还是最后一步手滑（执行差）？
- 技能级微调（Skill-level SFT）：既然知道 CUA 在 Housing 的 Filter 技能上弱于 Qwen，那就针对性地构造包含复杂筛选条件的训练数据，而不是泛泛地增加全站数据。
- 构建确定性评测环境：WEB STEP 证明，通过后端 MDP 驱动前端渲染，可以实现零人工成本的自动化过程评估。这是目前最可行的 Agent 诊断方案。
## 局限与展望WEB STEP 目前基于自托管的模拟网站（10 个站点），虽然覆盖了电商、邮件等场景，但与真实互联网的复杂性仍有差距。此外，语义 MDP 的构建依赖于对网站逻辑的先验建模，对于动态变化极大的真实网页，如何自动化生成这种“双生体”仍是挑战。
但这篇论文指明了方向： 未来的 Agent 评测，必须从“黑盒打分”走向“白盒诊断”。
