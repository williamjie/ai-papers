# ⭐⭐⭐½ 权重加密：开源模型如何实现能力分级访问

**日期**: 2026-06-23

论文 : Toward Open Weight Models Without Risks: Separating Public and Private Capabilities in LLMs链接 : https://arxiv.org/abs/2606.21638开源大模型（Open-Weight LLMs）面临一个核心矛盾：既要公开权重促进科研，又要防止敏感能力泄露。现有方案要么阉割模型导致“全员降智”，要么依赖闭源 API 丧失自主权。这篇论文提出了一种名为分层语言模型（Tiered Language Models, TLMs）的新范式： 用同一套权重文件，通过一个极小的“密钥”切换出不同的能力层级 。
### 痛点：为什么现有的“密码锁”方案行不通？
目前常见的做法是在 Prompt 中嵌入秘密 Token（如 Greenblatt et al., 2024）。这种做法本质脆弱。因为密钥存在于输入空间，攻击者可以通过少量演示数据或强化学习轻松绕过限制。另一种方案是使用 LoRA 适配器分发特定知识，但这破坏了“单一权重文件”的简洁性，且随着层级增加，管理大量额外参数变得极其繁琐。
TLMs 的核心 Insight 在于将授权机制从 输入空间 转移到 模型参数结构 本身。密钥不再是一串文本，而是一个针对少量参数的排列置换（Permutation）指令。没有密钥，模型运行在默认的公开配置下；拥有密钥，用户通过重排部分权重构建替代计算图，从而解锁私有能力。
### 方法拆解：不对称联合预训练 + KL 正则化TLMs 并非简单的后处理技巧，其有效性依赖于精心设计的两阶段训练协议：
-不对称联合预训练（Asymmetric Joint Pretraining）
模型被划分为“层级参数”（Tier Parameters, SS，约占 5%）和“互补参数”（Complementary Parameters, Sˉ\bar{S}ˉ）。
公开配置（Public Config）仅通过 Sˉ\bar{S}ˉ 接收梯度。
- 密钥配置（Keyed Config）通过 SS 和 Sˉ\bar{S}ˉ 接收梯度。
- 直觉：迫使公开模型学会不依赖那些即将被“加密”的参数，从而在后续微调中保持稳定性。
-私有微调与正则化（Private Fine-tuning with Regularization）
冻结 Sˉ\bar{S}ˉ，仅通过密钥配置更新 SS 以学习私有数据。同时引入 KL 散度正则项，惩罚公开配置行为偏离预训练状态的情况。这确保了即使共享权重被修改，公开模型的行为也不会发生灾难性遗忘或泄露。
### 关键结果：零泄露与极低开销论文在 180M 和 650M 参数规模的模型上进行了验证，结果令人印象深刻：
- 能力隔离：在西班牙语文本微调中，密钥配置的损失稳步下降，而公开配置在西班牙语上的表现毫无改善，证��私有知识未泄露。
- 指令跟随：密钥配置在 AlpacaEval 上的胜率从 ~50% 飙升至 >85%，而公开配置保持在 ~15% 的基础水平。
- 事实记忆：针对 400 条合成传记，密钥配置达到 100% 精确匹配，公开配置始终为 0%。

### 工程启示与安全边界对于工程实践，TLMs 提供了一种优雅的“单文件多租户”解决方案。它允许企业在同一权重文件上部署不同权限等级的服务，无需维护多个模型变体。
安全性测试显示：
- 抗微调提取：攻击者即使拥有部分私有数据并对公开配置进行全参数微调，也无法恢复被锁定的知识（准确率始终为 0）。
- 抗部分密钥泄露：仅知道 90% 的密钥排列，模型准确率依然接近零。只有当密钥完整度超过阈值时，能力才会突然显现，表现出类似密码学的“硬门槛”特性。
### 局限与展望尽管前景广阔，TLMs 目前仍面临挑战。实验仅在 180M-650M 规模进行，前沿大模型（Frontier LLMs）的干扰模式可能不同。此外，私有微调会在权重幅度上留下统计指纹，攻击者可能通过白盒分析识别层级参数。未来工作需探索更复杂的混淆策略及更大规模的验证。
总之，TLMs 为开源模型的访问控制提供了新的设计空间： 不是 withholding weights，而是 reconfiguring them 。这或许是实现“安全开源”的重要一步。
