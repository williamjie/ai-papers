他们提出了 PhoneHarness，一个支持混合动作空间的执行框架及配套基准测试。核心观点很硬核：不要把所有任务都简化为屏幕导航，Agent 必须具备“何时用 CLI、何时用 GUI、何时调 API”的决策能力，并且要能留下可审计的执行痕迹。
## 为什么纯 GUI Agent 走不通了？
现有的 Mobile-Agent、AndroidWorld 等工作主要解决的是“观察截图-点击坐标”的问题。但在真实场景中，很多任务根本不需要看图。
比如“查询电影信息并发送邮件”，这涉及 App 内浏览、外部检索、文本处理和状态变更。如果强行让 Agent 通过点击屏幕来完成所有步骤，不仅效率极低，而且极易因为 UI 变动或广告弹窗而失败。
PhoneHarness 指出，手机是一个充满权限和状态的复杂环境。可靠的自动化依赖于动作表面的路由（Action-Surface Routing）和可验证的执行结果，而不仅仅是视觉控制。
## 核心设计：确定性优先的路由机制PhoneHarness 的设计直觉非常工程化： 能用确定性的命令，绝不依赖脆弱的视觉交互。
它采用 Host-Device 架构，设备端运行 Agent 循环，主机端提供模型路由、GUI 代理和 MCP 风格工具服务。其动作空间分为三种模式：
- GUI/CLI 替代：如果任务既可以通过 App 界面完成，也可以通过 Shell/ADB 命令完成，优先选择更可靠的 CLI 路径。
- GUI 为主 + CLI 辅助：核心流程在 App 内，但利用 CLI 获取设备状态、准备文件或减少不稳定的屏幕导航。
- 纯 GUI 回退：只有当没有结构化路径可用时，才委托给视觉控制器进行有限的 GUI 交互。
这种“渐进式技能披露”（Progressive Skill Disclosure）机制也很关键。系统 Prompt 只包含技能索引，Agent 需要特定能力时才调用加载工具获取指令。这避免了上下文窗口爆炸，同时保持了动作空间的灵活性。
## 实验结果：混合路由带来的显著增益在 PhoneHarness Bench 的 124 个标注任务上，PhoneHarness 的表现证明了其设计的有效性。

更有趣的发现在于任务类型的分布。在“设备/系统操作”和“工具辅助工作流”上，PhoneHarness 优势巨大。但在纯视觉的单 App GUI 任务上，Seed2.0-Pro (76.7%) 甚至略高于 PhoneHarness (63.3%)。
⚠️ 反直觉发现 ：PhoneHarness 并不是一个“更好的点击器”，而是一个“更聪明的调度器”。它的增益来自于在可路由的任务上避开了脆弱的 GUI 交互，而不是提升了视觉识别能力。
在执行效率上，PhoneHarness 平均每个任务仅需 23 步 ，低于 Seed2.0-Pro 的 24 步和 AutoGLM-Phone 的 37 步。这说明引入 CLI 和工具不仅提高了成功率，还减少了无效探索。
## 工程启示：Agent 架构的新方向这篇论文对实际落地有几个重要指导意义：
- 混合动作空间是标配：未来的手机 Agent 不应局限于 GUI 控制。集成 ADB、Shell 命令和 MCP 工具接口，能显著提升鲁棒性。
- 可审计性至关重要：PhoneHarness 强调 Trace-backed Verification。每个步骤都要留下证据（如发送的邮件、修改的设置），这不仅用于评测，更是生产环境中调试和安全审计的基础。
- 模型分工协作：实验显示，使用 DeepSeek V4 Flash 作为控制器，搭配 Seed2.0-Pro 作为 GUI 工人，整体通过率最高 (74.8%)。这表明“文本强模型做规划 + 视觉强模型做执行”的异构架构是有效路径。
## 局限与展望目前 PhoneHarness Bench 仍有局限性。真实 App 环境比模拟环境更脆弱，网络波动、登录状态和权限弹窗都会影响任务可行性。此外，安全性评估仍处于探索阶段，虽然引入了拒绝敏感操作的机制，但距离完全可靠的安全认证还有距离。
不过，PhoneHarness 提出的“虚拟显示支持”方向很有潜力：让 Agent 在后台独立显示屏上工作，而不干扰用户前台操作。如果这一技术成熟，手机 Agent 将从“演示型控制”真正走向“并发式辅助”。
对于工程师而言，别再死磕 OCR 和点击坐标了。给你的 Agent 一把 CLI 钥匙，让它学会走捷径，这才是手机自动化的未来。
