论文 : EnterpriseClawBench: Benchmarking Agents from Real Workplace Sessions链接 : https://arxiv.org/abs/2606.23654现在的 LLM 评测大多还在卷“做题”，但企业级 Agent 的核心战场是 工作空间（Workspace） 。这篇论文提出的 EnterpriseClawBench，直接从真实企业内部会话中挖掘出 852 个可复现任务，揭示了当前 Agent 落地的最大痛点： 不是模型不够强，而是 Harness（代理框架）与模型的兼容性极差。
### 为什么现有评测不够用？
大多数基准测试（如 Workspace-Bench, EnterpriseBench）要么是人造数据，要么是模拟环境。这导致两个严重脱节：
- 真实性缺失：真实企业任务包含大量私有文件、复杂上下文和非结构化输入，人造数据无法覆盖这种噪声。
- 维度单一：传统评测只给一个总分，忽略了成本、运行时、多模态交付质量以及 Harness-Model 组合效应。
论文指出，企业 Agent 的成功不仅取决于“回答正确”，更取决于能否在持久化 Linux 沙箱中恢复输入、保持状态并生成可用的业务工件（Artifacts）。
### 核心 Insight：Harness 决定上限这篇论文最反直觉的发现是： 同一个模型，换不同的 Harness，性能可能天差地别。
作者构建了从原始会话到基准任务的自动化流水线。他们发现，Claude 系列模型在 Hermes 框架下表现大幅下滑。
- 现象：Sonnet 4.6 在 Claude Code/DeepAgents/OpenClaw 下得分稳定在 0.62–0.64。
- 崩塌：在 Hermes 下，同一模型分数暴跌至 0.458。
- 原因：Hermes 倾向于通过审批检查阻断环境探测和脚本执行，或将工作路由到子任务，导致执行轨迹被截断，最终未能生成稳定文件。
⚠️ 关键结论 ：这不是模型能力差距，而是 Harness-Model 兼容性（Compatibility） 问题。选择错误的框架，会让最强模型变成“废柴”。
### 实验数据拆解在人工审计的 120-task Lite 子集上，最佳组合（Codex + GPT-5.5）得分仅为 0.663 。这说明企业任务远未饱和。
维度 关键发现 数据支撑 成本收益 边际效应递减 低成本低分，中段提升明显，高成本区呈对数增长曲线。Hermes/Claude 组合是高成本的典型“异常值”。 角色差异 通用 vs 垂直 GPT-5.5 是最稳健的通用选手。 营销和财务/运营 类任务最难，得分普遍较低，因涉及大量公司特定惯例。 交付类型 格式影响排名 GPT-5.5 在 HTML/Code/JSON 上最强；Opus 4.6 在电子表格上表现最好。视觉判分存在系统性通胀。 技能迁移 创作者能力至关重要 技能注入方差极大。GPT-5.5 作为技能创作者（Creator），平均提升 +0.068 ；Haiku 4.5 作为创作者，平均反而下降 -0.094 。
### 工程启示：如何落地？
- 不要只选模型，要选组合：在部署前，必须测试 Harness-Model 矩阵。对于依赖环境交互的任务，避免使用过度限制脚本执行的框架。
- 多维评估是必须的：单一分数会掩盖真相。你需要同时监控：
硬性规则（Hard Rules）：文件是否生成、是否非空、可打开。
- 语义质量：基于文本和视觉的五个维度（准确性、相关性、深度、实用性、沟通）。
- 技能工程（Skill Engineering）需谨慎：论文显示，技能迁移高度依赖“创作者”模型的质量。弱模型生成的技能不仅无效，还可能污染强模型的执行路径。建议仅使用最强模型蒸馏技能。
- 视觉评测不可信：人类与 LLM 在视觉工件（PPT、图片）上的评分相关性为负（Spearman -0.259），而在文本上相关性较高（0.790）。目前的多模态裁判仍需人工校准。
### 局限与展望由于数据来自单一企业内部，基准可能无法完全代表所有行业。且数据未开源，限制了社区复现。但更重要的是，它指出了一个方向： Agent 评测应从“模型中心”转向“系统中心” ，关注整个工具链的协同效应。
对于工程师而言，这篇论文的价值不在于那个 0.663 的分数，而在于它警告我们：在真实工作流中， 框架的选择可能比模型版本更重要。
