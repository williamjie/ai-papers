论文 : MobileForge: Annotation-Free Adaptation for Mobile GUI Agents with Hierarchical Feedback-Guided Policy Optimization链接 : https://arxiv.org/abs/2606.19930现在的 Mobile GUI Agent 普遍面临一个尴尬局面：通用大模型（如 Qwen-VL）虽然聪明，但面对具体的 App 交互时往往“眼高手低”。要让它变强，传统的做法是搞大量的人工标注数据或专家演示。但这在移动端简直是天方夜谭——App 更新太快、长尾功能太多，人工标注的成本高到离谱且极易过时。
这篇来自快手和浙江大学的论文提出了一个非常务实的解法： MobileForge 。它不再依赖昂贵的人类标注，而是通过一套“自动探索-生成任务-自我纠错”的闭环系统，让 Agent 在真实 App 中自己练级。
## 现有方案的痛点：缺乏统一的适配底座目前的无监督 GUI 学习（Annotation-free learning）虽然减少了人工干预，但存在两个核心瓶颈：
- 数据与评估脱节：生成的任务往往缺乏真实的 App 交互支撑，导致训练数据和实际运行时的反馈是割裂的。
- 优化信号粗糙：大多数方法把整个 rollout（轨迹）当成一个孤立事件，只给最终成败的稀疏奖励。这种粗粒度的反馈很难转化为可靠的策略改进信号，Agent 很难从失败中吸取教训。
## 核心设计拆解：MobileGym + HiFPOMobileForge 的核心创新在于构建了一个完整的适配闭环，主要由两部分组成： MobileGym （交互与评估底座）和 HiFPO （分层反馈引导的策略优化）。
### 1. MobileGym：让数据“落地”
它不是简单的爬虫，而是一个功能感知的探索器。
- 真实探索：利用 APK 的结构信息和截图进行深度优先遍历，记录真实的 UI 状态流转。这保证了生成的任务是“可执行”的，而不是凭空捏造的幻觉。
- 分层评估（Critic）：这是关键。它不仅仅判断任务成功与否（Outcome），还评估每一步的操作合理性（Step-level feedback），并生成纠正性提示（Corrective Hints）。
### 2. HiFPO：把失败变成经验这是论文最精彩的工程直觉所在。传统的 RL 往往直接丢弃失败的轨迹，但 HiFPO 认为： 失败中包含了极具价值的局部正确决策和避坑指南。
- 多轮尝试与提示累积：对同一个任务进行多次尝试。如果第一次失败了，Critic 生成的“纠正性提示”会被作为上下文（Context）喂给下一次尝试。
- 去伪存真的过滤机制：
剔除 mastered 任务：如果所有尝试都成功了，说明模型已经掌握，无需再练。
- 保留失败与部分成功：即使全部失败，只要其中包含合理的局部步骤（Reasonable steps），就会被提取出来用于训练。
- Hint-Contextualized GRPO：这是训练的核心。它使用分组相对策略优化（Group Relative Policy Optimization, GRPO）。在计算奖励时，不仅看当前截图，还结合之前的“纠正性提示”。这意味着模型是在学习：“在已知之前犯了这些错的前提下，下一步该怎么走才是最优的。”
⚠️ 反直觉发现 ：实验表明，直接对无标注数据进行监督微调（SFT）效果很差，甚至不如基线。只有结合分层反馈和 GRPO 的优化，才能将自动生成的数据转化为有效的能力提升。
## 关键结果：逼近闭源专用模型MobileForge 的效果非常硬核，特别是在“零人工标注”的前提下。

- 通用模型逆袭：经过 MobileForge 适配的 Qwen3-VL-8B，在 AndroidWorld 上的 Pass@3 达到了 69.0%，几乎追平了专门针对 GUI 优化的闭源基线模型 GUI-Owl-1.5-8B。
- 最强开源 Agent：基于更强的 GUI-Owl 底座训练出的 ForgeOwl-8B，在 AndroidWorld 上达到 77.6%，并在跨域测试 MobileWorld 上以 41.0% 的成功率成为当前最强的开源 Mobile GUI Agent。
- 消融实验验证：移除“纠正性提示”后，整体成功率从 77.0% 暴跌至 52.0%。这证明了多轮尝试中的反馈累积是提升性能的关键，而不仅仅是增加采样次数。
## 工程启示- 失败数据比成功数据更值钱：在 Agent 训练中，不要简单丢弃失败的轨迹。通过细粒度的步骤评估（Step-level assessment），你可以从失败中提取出正确的局部动作，用于强化模型的基础 UI 理解能力。
- 提示作为上下文而非奖励：将 Critic 的反馈转化为 Prompt 中的 Context（如“上一步你点错了，因为…”），比单纯调整 Reward Score 更有效。这符合 LLM 的特性，它更擅长从文本指令中学习，而不是从标量奖励中反向传播。
- GRPO 是 GUI Agent 的微调利器：相比于传统的 PPO 或 SFT，基于分组的 GRPO 不需要训练 Critic 模型，且能更好地处理多模态动作的离散奖励问题，非常适合 GUI 这种结构化输出场景。
## 局限与展望MobileForge 目前仍受限于探索阶段的 App 生态覆盖范围。对于跨 App 的长链路任务、游戏类交互以及需要复杂记忆的任务，提升效果有限。此外，自动评估器（Critic）的质量直接决定了训练上限，未来如何构建更可靠、低成本的自动化 Critic 仍是工程落地的关键挑战。
