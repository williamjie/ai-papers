# ⭐⭐⭐ AI Agent 身份认证：像律师出庭一样严谨

**日期**: 2026-06-23

---

原文 : Agent Auth: A lawyer’s day in court来源 : https://www.cncf.io/blog/2026/06/23/agent-auth-a-lawyers-day-in-court/将 AI Agent 视为“微服务 Plus”是一个极具洞察力的视角。
它继承了传统微服务的所有基础设施需求，但在身份认证、策略执行和可观测性上提出了更严苛的要求。
这篇文章通过一个生动的法律类比，揭示了 Agent 架构中容易被忽视的核心痛点： 代理关系的明确性与可验证性 。
### 为什么传统认证模型失效了？
在传统的单体或微服务应用中，服务通常以自己的身份运行，或者代表单一用户上下文进行操作。
但在 AI Agent 场景中，情况变得复杂得多：
- 多重代理：一个 Agent 可能同时为成千上万的不同用户执行任务。
- 行为不可预测性：LLM 的非确定性要求更严格的护栏（Guardrails）和策略强制。
- 上下文复杂性：我们需要审计的不仅是请求结果，还包括 Prompt、工具调用链以及响应内容。
如果继续沿用简单的 API Key 或单一 JWT 令牌，我们将面临巨大的安全盲区：系统无法区分是 Agent 自己在操作，还是它在代表用户操作，更无法验证这种代表权是否被滥用。
### “律师出庭”模型：拆解 Agent Auth 核心逻辑作者引入了一个精妙的类比： AI Agent 就像法庭上的律师 。
这个类比完美映射了 Agent 认证所需的三个关键要素：
法律场景 Agent 技术对应 核心含义 律师身份 Agent Identity 系统必须知道是哪个 Agent 在发起请求。这是信任的基础。 委托人身份 Principal Identity 系统必须知道 Agent 代表哪个用户（Alice）。 授权文件 Delegation Token (OBO) 证明律师有权代表 Alice 处理特定案件的凭证。
⚠️ 关键洞察 ：拥有有效的委托令牌（Delegation Token）并不等于拥有无限权限。
就像法官在核实律师身份和委托书后，仍需判断该律师是否被允许在 当前案件 中执行 具体动作 一样，Agent 系统必须引入 策略执行点（Policy Enforcement Point） 。
这意味着认证流程分为两步：
- 身份验证与委托验证：你是谁？你代表谁？你有代理权吗？
- 授权与策略检查：你被允许对资源 X 执行动作 Y 吗？这是否符合当前的安全策略和范围（Scope）？
### 架构选型：为何需要 AI Native Gateway？
如果让每个 Agent 独立实现身份传播、委托验证和审计日志，不仅开发成本高昂，而且极易出错。
解决方案是将这些横切关注点下沉到基础设施层，即 AI Native Gateway 和 Service Mesh 。
在这个架构中，Gateway 和 Mesh 扮演了“法院书记员”、“法警”和“档案室”的角色：
- 建立强 Agent 身份：利用 SPIFFE 等技术为每个 Agent 分配唯一的、可验证的身份标识。
- 跨请求携带 Principal 身份：确保用户上下文在整个调用链中不丢失。
- 签发与验证委托令牌：处理 On-Behalf-Of (OBO) 令牌的生成和校验。
- 强制执行政策与范围：在流量入口处拦截违规操作。
- 提供可观测性与审计追踪：记录所有 Agent 行为，确保事后可追溯。
通过结合 cert-manager 、 Istio 等现有云原生组件，我们可以构建一个平台，让 Agent 专注于业务逻辑（如生成代码、分析数据），而将身份治理交给平台处理。
### 工程启示与局限思考对于正在构建 AI 应用的云原生团队，这篇文章提供了明确的技术指引：
- 不要混淆 Agent 与 User：在数据库设计、日志记录和权限模型中，必须严格区分操作者（Agent）和受益者（User）。
- 基础设施先行：在编写第一个 Agent 之前，先规划好身份传播和委托机制。 retrofitting（后期改造）认证逻辑的成本极高。
- 利用现有生态：无需从零造轮子，SPIFFE/SPIRE 和 Istio 已经提供了强大的身份和策略执行能力，关键在于如何适配 AI 特有的 OBO 场景。
当然，这个模型也有其边界。它主要解决了 授权 层面的问题，对于 LLM 幻觉导致的内容安全风险，仍需结合其他护栏技术（如输出过滤、内容审核）共同解决。
总之，Agent Auth 的核心在于 显式化、可验证、可执行 的代理关系。只有像法庭一样严谨地对待每一次“代理”行为，AI 应用才能在生产环境中安全落地。
